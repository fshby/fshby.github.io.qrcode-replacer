## 一、核心目标

用户上传一张模板图（如海报、邀请函）和一张二维码图，工具自动（或通过简单交互）将二维码无损、完美地替换到模板中指定的区域，支持透视变形、旋转、缩放，最终导出合成图片。同时提供用户支持渠道（捐赠+联系入口），提升用户体验和开发者收益。

## 二、技术框架

### 2.1 整体架构（纯前端，无需后端）

```
用户浏览器
├── 上传模块（拖拽/点击上传模板图 + 二维码图）
├── 图像处理引擎（Canvas + 透视变换库）
│   ├── 二维码区域定位（自动检测 / 手动标记）
│   ├── 透视变换计算（单应性矩阵）
│   └── 像素级合成（双线性插值 + 边缘融合）
├── 交互层（fabric.js 实现锚点拖拽、实时预览）
├── 通知层（Toast组件实现非阻塞式反馈）
├── 支持层（捐赠入口 + 开发者联系入口）
└── 导出模块（toBlob / toDataURL 输出 PNG/JPG）
```

### 2.2 关键组件与技术选型

| 模块 | 技术/库 | 体积 | 作用 |
|------|---------|------|------|
| 画布交互 | fabric.js | ~300KB | 支持图形拖拽、缩放、旋转，锚点操作 |
| 二维码定位（自动） | jsQR | ~50KB | 检测模板图中已有的二维码，返回角点坐标 |
| 透视变换 | perspective-transform | <5KB | 计算4点到4点的投影矩阵，做图像扭曲 |
| 图像解码与合成 | 原生 Canvas API | 0 | 读取像素、双线性插值、合成导出 |
| 用户界面 | Vue 3 / React + Tailwind CSS | 按需 | 轻量 SPA，响应式布局 |
| 通知组件 | Toast（自定义） | ~5KB | 非阻塞式用户反馈 |

总包大小控制在 500KB~1MB（不含 UI 框架），完全适合静态托管。

### 2.3 核心算法流程

```
1. 加载模板图 → 显示在 Canvas 上
2. 尝试自动检测：
   - 调用 jsQR.scan(canvas) 获取二维码角点坐标
   - 若成功 → 直接进入步骤4
   - 若失败 → 进入手动模式
3. 手动模式：用户在模板图上点击/拖拽四个锚点，标记二维码区域
4. 加载二维码图（保持原始分辨率）
5. 计算透视变换矩阵：
   - 源四边形：二维码图的四个顶点 (0,0)-(w,0)-(w,h)-(0,h)
   - 目标四边形：步骤2/3得到的四个角点坐标
   - 使用 perspective-transform 生成变换对象
6. 逐像素合成：
   - 遍历目标区域 bounding box 内的每个像素
   - 逆变换得到源图对应坐标
   - 双线性插值取色
   - 写入目标 Canvas
7. 可选边缘融合（圆角/羽化）
8. 导出合成图（保留模板原始分辨率）
9. 显示感谢提示（捐赠入口）
```

### 2.4 性能指标定义

| 指标 | 目标值 | 备注 |
|------|--------|------|
| 最大上传尺寸 | 2048×2048px | 超出时自动缩放至该尺寸 |
| 二维码最小分辨率 | 200×200px | 低于此值提示用户更换高清二维码 |
| 自动检测响应时间 | < 500ms | 复杂图像不超过1秒 |
| 透视变换合成时间 | < 300ms（1000×1000区域） | 现代浏览器 |
| 导出响应时间 | < 200ms | PNG格式 |
| 内存占用 | < 100MB | 避免浏览器崩溃 |
| Toast通知显示 | < 100ms | 动画流畅无卡顿 |

### 2.5 错误处理流程

| 错误类型 | 触发条件 | 用户提示文案 | 处理策略 |
|----------|----------|--------------|----------|
| 格式不支持 | 上传非图片文件（非PNG/JPG/JPEG/WEBP） | "请上传图片文件（支持PNG、JPG、WEBP格式）" | 拦截上传，保留上传区域状态 |
| 尺寸超限 | 图片边长超过2048px | "图片尺寸过大，已自动缩放至2048px以内" | 自动缩放至最大尺寸，继续流程 |
| 分辨率不足 | 二维码图片边长低于200px | "二维码分辨率过低，可能导致合成后无法扫描，请上传更高清晰度的二维码" | 允许继续，但高亮警告 |
| 检测失败 | 自动检测未发现二维码 | "未检测到二维码，请手动标记四个角点" | 自动切换至手动模式，显示锚点 |
| 合成失败 | 透视变换计算出错 | "合成过程中出现错误，请重试" | 清空画布，重新上传 |
| 解码验证失败 | 合成后二维码无法被扫描 | "合成后的二维码无法扫描，请调整标记区域或更换二维码" | 保留合成结果，提示用户调整 |
| 内存溢出 | 浏览器内存不足 | "当前页面内存不足，请刷新后重试" | 提示刷新页面 |
| 网络异常 | 图片加载失败 | "图片加载失败，请检查网络或重新上传" | 重新触发上传流程 |

### 2.6 UI/UX设计规范

#### 2.6.1 整体布局

页面结构分为三大区域：

顶部导航区（高度60px）：Logo + 标题 + 捐赠入口（❤️ 请喝杯咖啡）

中部操作区（flex-1）：

左侧：上传面板（模板图上传框 + 二维码图上传框 + 使用说明 + 捐赠提示）

右侧：Canvas预览区（实时显示合成效果）

底部工具栏（高度80px）：操作按钮组

右下角浮动按钮：☕ 捐赠入口（常驻显示）

#### 2.6.2 上传面板设计

模板图上传框：

- 初始状态：虚线边框，灰色背景，显示上传图标和"点击或拖拽上传模板图"文字

- 悬停状态：边框变为蓝色，背景变亮

- 已上传状态：显示缩略图，右上角有关闭按钮

二维码图上传框：

- 设计同模板图上传框

- 提示文字："点击或拖拽上传二维码图"

- 下方显示最小分辨率提示："建议分辨率≥200×200px"

#### 2.6.3 Canvas预览区设计

- 画布居中显示，最大尺寸占预览区90%

- 自动检测成功时：用半透明蓝色边框高亮标记二维码区域，四角显示可拖拽锚点

- 手动模式时：显示四个初始锚点（位于画布四角附近），用户可拖拽调整

- 合成完成后：显示最终效果，保留锚点供用户微调

- 缩放控制：底部提供缩放滑块（50%-200%），支持滚轮缩放

#### 2.6.4 工具栏设计

按钮组从左到右依次排列：

- 重置按钮：恢复初始状态

- 重新检测按钮：重新加载原始模板并检测二维码

- 导出按钮：导出PNG图片（主操作按钮，高亮显示）

#### 2.6.5 Toast通知设计

| 类型 | 颜色 | 图标 | 场景 |
|------|------|------|------|
| success | 绿色 | ✅ | 操作成功（上传、检测、合成、导出） |
| error | 红色 | ❌ | 操作失败、错误提示 |
| warning | 黄色 | ⚠️ | 警告信息（分辨率不足等） |
| info | 蓝色 | ℹ️ | 信息提示（加载中、操作指引） |

- 自动消失时间：3秒

- 支持手动关闭

- 支持多个通知堆叠显示

## 三、可行性分析

### 3.1 技术可行性 ✅

| 维度 | 评估 |
|------|------|
| 算法成熟度 | 透视变换是计算机视觉经典基础，开源库稳定可靠 |
| 性能 | 纯 Canvas 逐像素运算，1000×1000 区域耗时 < 200ms（现代浏览器） |
| 兼容性 | 所有主流浏览器（Chrome/Firefox/Safari/Edge）均支持 Canvas 2D |
| 精度 | 双线性插值可达到亚像素级，肉眼几乎看不出变形失真 |
| 无损保障 | 二维码图以原始分辨率参与计算，不提前缩放；导出格式可选 PNG 无损 |
| Toast通知 | 纯CSS动画实现，无性能开销 |

### 3.2 用户体验可行性 ✅

- 自动模式：用户只需上传两张图，一键完成替换，学习成本极低。
- 手动模式：四个锚点拖拽，实时预览，符合设计师习惯。
- 降级策略：自动检测失败后平滑切换到手动，不打断流程。
- 非阻塞反馈：Toast通知不打断用户操作，提升流畅度。
- 捐赠入口：多触点设计，在用户完成导出时自然露出，降低抵触感。

### 3.3 业务可行性 ✅

| 因素 | 说明 |
|------|------|
| 差异化 | 市面工具（草料、Canva）不支持上传替换+透视自适应 |
| 目标用户 | 社群运营、微商、线下活动组织者，高频制作群码海报 |
| 变现潜力 | 免费版靠捐赠支持，付费版可解锁高清无水印+批量处理 |
| 部署成本 | 纯静态页面，GitHub Pages / Vercel / 腾讯云 COS 均可，月费≈0 |

### 3.4 风险与应对

| 风险 | 应对措施 |
|------|----------|
| 自动检测成功率低（复杂模板） | 始终提供手动模式兜底，并提示用户确保二维码三个定位图案完整可见 |
| 大图合成内存溢出（>4000px） | 限制最大上传尺寸（建议 4096px），或使用 OffscreenCanvas 分段处理 |
| 二维码替换后无法扫描 | 合成后自动调用 jsQR 验证解码结果，若失败则提示用户调整区域或提高二维码分辨率 |
| 微信环境限制 | 工具本身不涉及跳转，仅生成图片，无合规风险 |
| 用户对捐赠入口抵触 | 使用幽默文案降低抵触感，不强制弹窗，仅在导出完成后友好提示 |

## 五、技术实现示例（核心代码片段）

### 5.1 透视变换合成函数

```typescript
function compositeQR(templateCanvas, qrImage, srcCorners, dstCorners) {
  const ctx = templateCanvas.getContext('2d');
  const qrW = qrImage.width, qrH = qrImage.height;
  const src = [0, 0, qrW, 0, qrW, qrH, 0, qrH];
  const dst = dstCorners.flat(); // [x1,y1,x2,y2,x3,y3,x4,y4]
  const transform = perspectiveTransform(src, dst);
  const inv = transform.inverse();

  const xs = dst.filter((_, i) => i % 2 === 0);
  const ys = dst.filter((_, i) => i % 2 === 1);
  const minX = Math.min(...xs), maxX = Math.max(...xs);
  const minY = Math.min(...ys), maxY = Math.max(...ys);

  const imageData = ctx.getImageData(minX, minY, maxX-minX, maxY-minY);
  const qrData = getImageDataFromImage(qrImage);

  for (let y = minY; y < maxY; y++) {
    for (let x = minX; x < maxX; x++) {
      const [sx, sy] = inv.transform(x, y);
      if (sx >= 0 && sx < qrW && sy >= 0 && sy < qrH) {
        const color = bilinearInterpolate(qrData, sx, sy, qrW, qrH);
        const idx = ((y - minY) * (maxX - minX) + (x - minX)) * 4;
        imageData.data[idx] = color.r;
        imageData.data[idx+1] = color.g;
        imageData.data[idx+2] = color.b;
        imageData.data[idx+3] = 255;
      }
    }
  }
  ctx.putImageData(imageData, minX, minY);
}
```

## 六、结论
